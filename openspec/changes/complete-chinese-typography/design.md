## Context

当前 core 包位于 `packages/core/src/rules/`，已实现以下规则：
- `no-duplicate-punctuation`: 避免重复标点
- `no-space-around-fullwidth`: 全角标点周围不添加空格
- `no-space-between-number-unit`: 数字与单位之间不添加空格
- `space-around-block`: 块级元素周围空格
- `space-around-char`: 中英文/数字之间空格管理
- `strict-fullwidth-and-halfwidth`: 空文件，待实现

根据中文排版指南，还需实现：
- 全角/半角转换（标点、数字）
  - **半角转全角策略**：在中文语境下，将半角标点 `.,;:!?()[]{}` 转换为全角 `．，：；！？（）［］｛｝`
  - **数字转换策略**：中文段落中使用全角数字，英文/数字段落保留半角

## Goals / Non-Goals

**Goals:**
- 实现 1 个新规则：`strict-fullwidth-and-halfwidth`
- 使用 Vitest 为所有规则编写单元测试
- 测试覆盖率目标：所有导出的规则函数

**Non-Goals:**
- 不修改现有已实现规则的行为（保持向后兼容）
- 不实现 ESLint 规则的 fixable 功能（仅保持 transform 功能）
- 不修改 Prettier 插件或 ESLint 插件的实现

## Decisions

### 1. 测试框架选择 Vitest
**决定**: 使用 Vitest 而非 Jest
**理由**: Vitest 与项目使用的 bunchee 构建工具同属 Vite 生态，配置简洁、运行速度快，且支持 TypeScript 原生。

### 2. 规则导出结构
**决定**: 每个规则独立文件，通过 `rules/index.ts` 统一导出
**理由**: 现有代码已采用此模式，保持一致便于维护。每个规则文件包含：
- 主转换函数（如 `transform(text): string`）
- 规则选项类型（可选）
- 单元测试文件同目录下的 `*.test.ts`

### 3. 半角转全角转换策略
**决定**: 采用"中文语境检测"策略，仅在 CJK 字符附近应用转换
**理由**:
- 全角标点在纯英文环境中影响可读性
- 中文段落中混合半角标点不符合排版规范
**实现方式**:
- 定义 CJK 字符范围：CJK Unified Ideographs、punctuation
- 转换规则表：`,` → `，`、`.` → `．`、`:` → `：`、`;` → `；`、`!` → `！`、`?` → `？`、`(` → `（`、`)` → `）`、`[` → `［`、`]` → `］`、`{` → `｛`、`}` → `｝`
- 检测算法：遍历文本，当发现 CJK 字符时，对后续/周围的半角标点进行转换

### 4. 数字全角/半角策略
**决定**: 数字默认保留半角，仅在全中文句子中转换为全角
**理由**:
- 数字在全角模式下宽度是半角的 2 倍，影响排版美观度
- 纯中文句子（如"今天气温三十五度"）使用全角数字更协调
- 混合中英文时保留半角数字更实用
**实现方式**:
- 判断段落是否以中文开头或整体为中文语境
- 数字转换表：`0` → `０`、`1` → `１`、... `9` → `９`
- 特殊场景保留：度数 `90°`、百分比 `15%`、小数 `3.14`

## Risks / Trade-offs

- **[风险] 中文语境检测可能误判** → 当段落包含少量 CJK 字符时（如技术文档中的中文注释），可能错误转换英文标点
  - **缓解**：检测段落整体 CJK 字符比例，超过阈值才触发转换
- **[权衡] 全角/半角转换可能影响英文内容** → 仅在中文语境（检测到 CJK 字符）时应用转换
